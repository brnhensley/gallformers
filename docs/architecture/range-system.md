# Range System Architecture

How gallformers tracks where species occur geographically.

## Data Model

### Two Range Tables

| Table | Owner | Fields | Purpose |
|-------|-------|--------|---------|
| `host_range` | Host plants | `species_id`, `place_id`, `precision`, `distribution_type` | Where plants grow (from WCVP) |
| `gall_range` | Gall species | `species_id`, `place_id`, `precision` | Curated gall distribution |

Both have composite PK `(species_id, place_id)` — one row per species-place pair.

**Key difference**: `host_range` has `distribution_type` ("native" or "introduced") because WCVP tracks this. `gall_range` does not — it's purely curated geographic extent.

### Precision

Each range entry has a `precision` field: `"exact"` (subdivision-level) or `"country"` (country-level). Country-level entries are expanded to leaf subdivision codes for map display via `Places.batch_leaf_descendant_ids/1`. See Places hierarchy below.

### Gall Range Resolution Order

A gall's displayed range follows this logic (`Ranges.get_display_range_for_gall/1`):

1. If `gall_range` has entries for this gall → use them (curated range is source of truth)
2. If `gall_range` is empty → fall back to the **union of all hosts' native ranges** (introduced ranges excluded)
3. If the gall has no hosts either → empty range

The fallback queries `gallhost` directly (string table name, no schema import) to avoid a circular boundary dependency between Ranges and Galls.

### DisplayRange

`Ranges.DisplayRange` is the struct used for map rendering:

```
%DisplayRange{
  in_range: ["US-CA", "CA-AB"],        # exact subdivision codes
  inherited_range: ["US-TX", "US-NY"],  # expanded from country-level entries
  introduced_range: ["BS"]              # subset marked as introduced (hosts only)
}
```

- `in_range` and `inherited_range` are always disjoint (inherited minus exact)
- `introduced_range` is only populated when `compute_display_range/2` is called with `with_introduced: true` (used on admin host pages, not on public gall pages)

## Places Hierarchy

```
continent (XN = North America)
  └── country (US = United States)
        └── state/province (US-CA = California)
```

Stored in `place` + `place_hierarchy` tables. Hierarchy traversal uses recursive CTEs (`ancestor_ids/1`, `descendant_ids/1`, `leaf_descendant_ids/1`).

**Leaf places** are places with no children — states, provinces, and countries without subdivisions (e.g., Bahamas). Only leaf places are selectable for species ranges.

- `Places.list_places/0` returns leaf places only (for UI selectors)
- `Places.list_all_places/0` returns all places including parents (needed for code-to-ID lookups during WCVP sync — using `list_places/0` silently drops country codes)

The `code` field is the stable join key between the database and the PMTiles vector tiles used for map rendering.

### Map Tiles and Boundary Data

The visual map layer (PMTiles) is generated by the toolchain in `services/boundaries/`. See `services/boundaries/README.md` for the full pipeline: Natural Earth shapefiles → ogr2ogr → tippecanoe → PMTiles. The `code` property in tile features matches `place.code` in the database.

Key supporting files in that directory:
- `generate_tdwg_mapping.py` — generates `priv/repo/data/tdwg_to_places.json`
- `extract_places.sh` — generates `priv/repo/data/global_places.json` for DB seeding
- `extract_bounds.py` — generates `priv/repo/data/place_bounds.json` for map viewport
- `PLACE_REFERENCE.md` — canonical list of all places and tile coverage

## WCVP Integration

[WCVP](https://wcvp.science.kew.org/) (World Checklist of Vascular Plants) from Kew Gardens is the authoritative source for host plant distribution data.

### Data Flow

```
Kew Gardens WCVP (CSV files)
  ↓  mix gallformers.wcvp.download + mix gallformers.wcvp.build_db
Separate Postgres database (wcvp_names + wcvp_distributions tables)
  ↓  Wcvp.Lookup (read-only Ecto repo)
TDWG L3 botanical region codes per species
  ↓  Wcvp.Tdwg.convert_tdwg_codes/2
ISO place codes with precision
  ↓  Ranges.update_host_places/2
host_range table
```

### TDWG-to-ISO Mapping

WCVP uses [TDWG WGSRPD Level 3](https://www.tdwg.org/standards/wgsrpd/) botanical region codes (368 codes globally). These do not map 1:1 to our ISO 3166-2 place codes. The static mapping file `priv/repo/data/tdwg_to_places.json` handles the translation.

**Key characteristics of the mapping:**

- **368 TDWG L3 codes** map to **4,514 ISO place entries** (all at "exact" precision)
- **One-to-many**: A single TDWG code often maps to multiple ISO subdivisions (e.g., TDWG "BZL" → BR-PR, BR-SC, BR-RS)
- **Many-to-one (219 cases)**: Multiple TDWG codes can map to the same ISO place code. This happens because TDWG botanical regions don't align with political boundaries. Examples:
  - `CA-NL` (Newfoundland and Labrador) ← TDWG codes `LAB` (Labrador) + `NFL` (Newfoundland)
  - `CL-VS` (Valparaíso) ← 4 TDWG codes: `CLC`, `DSV`, `EAS`, `JNF`
  - `AU-TAS` (Tasmania) ← TDWG `MAQ` (Macquarie I.) + `TAS` (Tasmania)

The mapping is generated by `services/boundaries/generate_tdwg_mapping.py`, which uses a combination of hardcoded region-to-country mappings (for complex multi-country TDWG codes like "BLT" → Estonia, Latvia, Lithuania) and database lookups.

### Duplicate Place Entries in Sync

Because of the many-to-one TDWG mapping, the same ISO place code can appear in **both** the native and introduced distribution lists for a single species. This happens when WCVP records a species as native in one TDWG region and introduced in an adjacent one, but both TDWG regions map to the same ISO subdivision.

`convert_tdwg_codes/2` deduplicates within each list (`Enum.uniq_by/2`), but when the native and introduced lists are concatenated for DB insertion, a place_id can appear twice with different `distribution_type` values. Since `host_range` has a composite PK `(species_id, place_id)`, this causes a unique constraint violation.

**Resolution**: `Ranges.update_host_places/2` deduplicates entries by `place_id` before insertion, with native taking priority over introduced. This is the correct semantic: if TDWG says a species is native in one sub-region and introduced in an overlapping one, we record it as native.

### Sync Pipeline

Two sync paths:

1. **Bulk sync** via Host Range Review admin page (`/admin/host-range`): Loads reference data once (`Plants.load_sync_ref_data/0`), then processes hosts sequentially with cached TDWG lookup and place maps
2. **Individual sync** via Host Edit page: Computes a 6-bucket diff (`Plants.compute_powo_diff/3`) showing what would change, lets the curator cherry-pick changes

The 6-bucket diff compares current range vs WCVP across a state matrix:

| Current \ WCVP | Not present | Native | Introduced |
|:---|:---|:---|:---|
| **Not present** | (no-op) | add_native | add_introduced |
| **Native** | remove | agree | reclassify_to_introduced |
| **Introduced** | remove | reclassify_to_native | agree |

## Gall Range Curation

### Exclusion Model

There is no "exclusions" table. Exclusions are implicit:

- `gall_range` contains only **included** places
- Anything in host range but NOT in `gall_range` is implicitly excluded
- To exclude a place: delete it from `gall_range`
- To include a place: insert it into `gall_range`

The admin GallHost page (`/admin/gallhost`) shows three map colors:
- **Green**: place is in `gall_range` (included)
- **Red**: place is in host range but NOT in `gall_range` (excluded)
- **Inherited**: leaf codes expanded from country-level `gall_range` entries

### Confirmation

`gall_traits.range_confirmed` tracks whether a curator has reviewed the gall's range. It is:
- Set to `true` when a curator clicks "Save & Confirm Range" on the GallHost page
- Reset to `false` when any of the gall's hosts' ranges change (`Galls.invalidate_gall_ranges_for_host/1`)
- Bulk-confirmable via the Gall Range Review page (`/admin/gall-range`)

The public gall page shows a "Needs review" badge when `range_confirmed` is false.

## Boundary Dependencies

```
Gallformers.Galls ──depends on──▶ Gallformers.Ranges
Gallformers.Ranges ──depends on──▶ Gallformers.Places, Gallformers.Species
```

Ranges cannot depend on Galls (circular). The fallback path in `get_display_range_for_gall/1` queries the `gallhost` table directly via string table name to avoid importing Galls schemas.
