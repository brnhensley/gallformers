# WCVP migration from SQLite to Postgres

## Design — Approach C: Separate Postgres Database

Eliminate SQLite by moving WCVP data to a dedicated Postgres database on the existing gallformers-db cluster. Repo.WCVP stays as a separate repo, adapter changes from SQLite3 to Postgres.

### Why separate database (not schema in main DB)

- pg_dump/backups are per-database — no bloat in app backups or `make download-db`
- Build task owns full lifecycle (CREATE TABLE, COPY, CREATE INDEX) — no Ecto migrations for WCVP tables
- Clean separation: reference data vs application data
- Independent backup/restore lifecycle
- Trade-off accepted: two repos/connection pools (WCVP pool is 2 connections — negligible)

### Architecture

- **Repo.WCVP** stays, adapter changes to `Ecto.Adapters.Postgres`
- **Typed Ecto schemas** for `wcvp_names` and `wcvp_distributions` — replaces untyped maps with structs throughout the WCVP call chain
- **Lookup module**: schema-based queries, `like` → `ilike`, stronger return types
- **Build task**: rewrite from `Exqlite.Sqlite3` to Postgrex — per-line CSV processing (trim, split on pipe, pad values) then `COPY FROM STDIN`. Hardcoded column list replaces dynamic CSV header discovery.
- **S3 artifact**: pg_dump of the wcvp database (all environments restore from this)
- **Drop deps**: `ecto_sqlite3`, `ecto_sqlite3_extras`

### Build flow (runs on developer machine)

```
mix wcvp.download           → 85MB zip from Kew, extract CSVs
mix wcvp.build_db --upload  → process CSVs → COPY into local wcvp DB
                            → pg_dump → upload dump to S3
```

CSV processing is non-trivial: per-line String.trim, String.split("|"), pad/trim to match column count. This is why it runs locally, not on the app server.

### Environment setup

| Environment | Setup |
|------------|-------|
| **Dev** | `createdb wcvp` (once) → `pg_restore` from S3 dump, or build locally from CSVs |
| **Test** | `createdb wcvp_test` (once) → load fixture subset via SQL file |
| **Preview** | `pg_restore` via fly proxy from S3 dump |
| **Prod** | `pg_restore` via fly proxy from S3 dump |

### pg_dump version matching

Controlled: local Postgres version matches or exceeds Fly Postgres version. Navigated before (see fly_postgres_ops memory).

### Phase 2 (separate matter: 0ae0)

Worker machine pattern for automated refresh. Removes developer from the critical path.

## Implementation Plan

**Goal:** Replace the WCVP SQLite database with a dedicated Postgres database, introducing typed Ecto schemas and eliminating the ecto_sqlite3 dependency.

**Architecture:** Repo.WCVP adapter swap, new Ecto schemas for type safety, Lookup queries migrate to schema-based with Postgres idioms, build task rewrites from Exqlite to Postgrex COPY FROM STDIN, new restore task for dev/test setup from S3 artifact.

**Tech Stack:** Ecto + Postgrex (replacing ecto_sqlite3 + Exqlite), Postgres COPY protocol, pg_dump/pg_restore.

### Task 1: Ecto Schemas

**Files:**
- Create: `lib/gallformers/wcvp/wcvp_name.ex`
- Create: `lib/gallformers/wcvp/wcvp_distribution.ex`

**Behavior:**

`WcvpName` maps to `wcvp_names`. All 31 columns from the Kew CSV as string fields. Primary key: `plant_name_id` (string, not autogenerated). Two virtual fields: `native_distribution` and `introduced_distribution` (both `{:array, :string}`, default `[]`).

`WcvpDistribution` maps to `wcvp_distributions`. All 11 columns as string fields. Primary key: `plant_locality_id` (string, not autogenerated).

The 11 columns: `plant_locality_id`, `plant_name_id`, `continent_code_l1`, `continent`, `region_code_l2`, `region`, `area_code_l3`, `area`, `introduced`, `extinct`, `location_doubtful`.

Both schemas are read-only — no changesets.

**Testing:**

- Virtual fields default to empty list

**Notes:**
- Use `@primary_key {:plant_name_id, :string, autogenerate: false}` (non-standard PK)
- All fields `:string` — WCVP source data is all text
- No `timestamps()` — these are external reference data

- Return types match the updated behaviour

### Task 4: Build Task Rewrite

Depends on: Task 1 (column list from schemas)

**Files:**
- Modify: `lib/mix/tasks/gallformers/wcvp/build_db.ex` (complete rewrite)
- Test: `test/mix/tasks/gallformers/wcvp/build_db_test.exs` (rewrite for Postgres)

**Behavior:**

Replace Exqlite.Sqlite3 with Postgrex. The task:
1. Opens a direct Postgrex connection to the wcvp database (not through Repo.WCVP — the task manages its own connection)
2. Drops and recreates tables with hardcoded DDL (column names from WcvpName/WcvpDistribution schemas)
3. Reads CSV files line by line, applies per-line processing (trim, split on `|`, pad_values)
4. Streams processed rows into Postgres via `COPY FROM STDIN` (Postgrex supports this)
5. Creates indexes after bulk load
6. Inserts meta record (built_at timestamp)
7. If `--upload`: shells out to `pg_dump` to dump the wcvp database, uploads dump to S3

CLI changes:
- Remove: `--output PATH` (no file to write)
- Add: `--database` (database name, default "wcvp") or derive from Repo.WCVP config
- Keep: `--names PATH`, `--dist PATH`, `--upload`

Hardcoded DDL for wcvp_names:
```sql
CREATE TABLE wcvp_names (
  plant_name_id TEXT PRIMARY KEY,
  ipni_id TEXT,
  taxon_rank TEXT,
  ... (all 31 columns as TEXT)
)
```

Indexes (Postgres equivalents of current SQLite indexes):
```sql
CREATE INDEX idx_wcvp_names_taxon_name ON wcvp_names(lower(taxon_name) text_pattern_ops)
CREATE INDEX idx_wcvp_names_genus ON wcvp_names(genus)
CREATE INDEX idx_wcvp_names_family ON wcvp_names(family)
CREATE INDEX idx_wcvp_names_accepted ON wcvp_names(accepted_plant_name_id)
CREATE INDEX idx_wcvp_names_status ON wcvp_names(taxon_status)
CREATE INDEX idx_wcvp_dist_name_id ON wcvp_distributions(plant_name_id)
CREATE INDEX idx_wcvp_dist_area ON wcvp_distributions(area_code_l3)
```

Note: `COLLATE NOCASE` → `lower(taxon_name) text_pattern_ops` for case-insensitive prefix matching with ilike.

The `pad_values/2` function stays — it handles malformed CSV rows from Kew. COPY FROM STDIN is strict about column counts, so padding/trimming happens in Elixir before streaming to COPY.

pg_dump for --upload:
```bash
pg_dump --format=custom --no-owner --no-acl wcvp > /tmp/wcvp.dump
```
Then upload to S3 at `public/wcvp.dump` (replacing the old `public/wcvp.sqlite`).

**Testing:**
- Creates tables with correct columns in wcvp_test database
- Inserts all rows from fixture CSVs (9 names, 13 distributions)
- Handles synonyms, unplaced, varieties (existing test coverage)
- Meta table has built_at timestamp
- Indexes exist (query pg_indexes)
- pad_values still handles short/long rows

**Notes:**
- The task opens its own Postgrex connection — don't use Repo.WCVP (the task might be creating tables that don't exist yet)
- For COPY FROM STDIN, refer to Postgrex docs: `Postgrex.query` with COPY statement + streaming. May need to use the lower-level protocol.
- The header line in CSVs is skipped (Stream.drop(1)) — same as current
- CSV is pipe-delimited, NOT comma-delimited. COPY delimiter must be `|`
- Empty string padding (current behavior) maps to empty TEXT in Postgres — same semantics

### Task 5: Restore Task (new)

**Files:**
- Create: `lib/mix/tasks/gallformers/wcvp/restore.ex`

**Behavior:**

New mix task for restoring the wcvp database from the S3 pg_dump artifact. Used for dev setup and environment bootstrapping.

```bash
mix gallformers.wcvp.restore [--database wcvp] [--url S3_URL]
```

Steps:
1. Download pg_dump from S3 (`public/wcvp.dump`)
2. Shell out to `pg_restore --clean --if-exists --no-owner --no-acl -d wcvp /tmp/wcvp.dump`
3. Clean up temp file
4. Report row counts from restored tables

Default S3 URL: `https://gallformers-backups.s3.amazonaws.com/public/wcvp.dump`

This is the developer-facing command for "give me WCVP data." Equivalent to the current "just have the SQLite file" but for Postgres.

**Testing:**
- Difficult to test end-to-end (requires S3 access + pg_restore binary)
- Test the URL construction and argument parsing
- Integration testing is manual

**Notes:**
- `pg_restore` must be installed locally (comes with Postgres)
- `--clean --if-exists` drops existing tables before restoring — safe to run repeatedly
- For prod/preview, user runs pg_restore manually via fly proxy — this task is for local dev only

### Cutover Sequence

After all tasks are complete and tested:

1. Build WCVP locally: `mix wcvp.download && mix wcvp.build_db --upload`
2. Create wcvp database on Fly: `fly proxy` → `CREATE DATABASE wcvp`
3. Restore data: `pg_restore --no-owner --no-acl -d wcvp -h localhost -p <proxy_port> /tmp/wcvp.dump`
4. Deploy new code (Repo.WCVP now connects to Postgres)
5. Verify admin WCVP features work on prod
6. Same for preview
7. Remove `WCVP_DATABASE_PATH` from Fly secrets
8. Delete `wcvp.sqlite` from Fly volume (`/data/wcvp.sqlite`)
9. Optionally delete `public/wcvp.sqlite` from S3 (keep `public/wcvp.dump`)
